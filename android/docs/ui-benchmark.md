# Info Push Android UI 调研（风格 C：中间态）

更新时间：2026-02-19

## 1) 参考样本（7个）

> 目标维度：卡片、导航、筛选、状态反馈。

1. **Google News**（聚合资讯）
2. **Apple News / Apple News+**（编辑精选 + 个性化）
3. **Feedly**（重度订阅/情报流）
4. **Inoreader**（RSS + 自动化 + 监控）
5. **SmartNews**（本地/全球混合分发）
6. **Reddit**（社区信息流 + 交互反馈）
7. **X（Twitter）**（实时流 + 轻筛选）

## 2) 共性 UI 模式总结

### A. 卡片（信息展示）
- 主流做法：**单列卡片流**，每条只保留“标题 + 关键元信息 + 1个主动作”。
- 高信息密度时：通过二级展开（详情页/打开原文）而不是首页堆过多字段。
- 行为反馈：收藏/已读/已添加等状态即时可见，避免“操作成功但无变化”。

### B. 导航（信息架构）
- 主流做法：底部主导航 + 页面内二级导航（Tab/Chip）。
- 对“源很多”的场景：页面内采用**横向可滚动来源选择**或可搜索列表。
- 首页强调“当前上下文”：例如“当前频道/当前源”显式标记。

### C. 筛选（发现与管理）
- 搜索入口尽量就近放置（列表顶部），支持名称/URL/标签等基础匹配。
- 推荐候选（AI发现）与本地已存在项需要**去重态**（禁用按钮、已添加标签）。
- 筛选优先轻量：关键词过滤 + 基础开关，复杂规则后续迭代。

### D. 状态反馈（loading/empty/error）
- 主流做法是统一状态语义与文案：
  - Loading：简短进行中提示
  - Empty：告诉用户“为什么空 + 下一步做什么”
  - Error：明确失败原因并可重试
- 状态组件复用，减少页面风格漂移。

## 3) 可直接落地本项目的设计决策（非空泛）

### 决策 D1：首页信息源选择可持续
- **做法**：将 Feed 当前选中 sourceId 写入本地偏好（Room preferences），重启恢复。
- **原因**：符合“连续消费”习惯，减少重复点击。
- **验收**：切换源后退出重进，仍定位原源（若源被禁用/删除则回退首个可用源）。

### 决策 D2：AI 搜索结果增加“已添加”去重态
- **做法**：用本地 sources.url 集合标记候选项；已存在时按钮禁用并显示“已添加”。
- **原因**：减少误操作与困惑，符合主流“不可重复关注/订阅”体验。
- **验收**：本地已存在 URL 的候选项不可再次添加，且有明确视觉状态。

### 决策 D3：信息源管理页支持滚动 + 搜索过滤
- **做法**：改为 `LazyColumn`；新增过滤输入框，匹配 name/url/tags。
- **原因**：源数量增长后可维护性显著提升，降低“找不到源”的操作成本。
- **验收**：长列表不截断；输入关键词实时过滤；空过滤结果有提示。

### 决策 D4：统一 Feed/Sources 的加载/空/错样式
- **做法**：抽取公共组件 `FeedbackSection` + `LoadingState` + `EmptyState` + `ErrorState`。
- **原因**：页面一致性提升，减少后续维护成本。
- **验收**：Feed/Sources 均通过同一套组件表达状态。

## 4) 与“风格 C（中间态）”的对应

- 不追求极简（信息不足）也不走重运营堆叠（视觉噪音过大）。
- 采取：
  - 首页保留核心操作（切源/刷新/打开原文）
  - 管理页增加必要筛选与明确状态
  - 避免引入复杂多级筛选器、复杂动画与过度装饰

## 5) 后续 P2 建议（本轮不阻塞）

1. 统一卡片动作区（打开/收藏/标记已读）为同一交互语义。
2. Source 维度统计（最近更新、文章数、错误计数）用于排序。
3. 空状态增强（引导去 AI 搜索或一键导入 OPML/JSON）。
4. 增加“按类型/标签”Chip筛选与排序（字母/最近更新）。
